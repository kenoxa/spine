# Permission Profile

Every queued task runs under a restricted permission profile enforced by **two layers**. Either layer alone has known gaps — both are required for v1.

## Layer 1 — Flag Gate (structural, weak)

`claude -p --allowed-tools '...' --disallowed-tools '...'` restricts which tools the child may invoke. Works well for coarse categories (`Bash` on or off, `Edit` on or off, `Task` on or off). **Does not reliably block subcommand patterns** — preflight #1 of 2617 showed `--disallowed-tools 'Bash(git push*)'` does not intercept `git push origin main`. Tool-category gate only.

## Layer 2 — Hook Gate (behavioral, authoritative)

`hooks/guard-queue-shell.sh` is a `PreToolUse` hook registered via `--settings <profile.json>`. It inspects the tool input, matches against the `patterns.deny` list, and emits a `permissionDecision: deny` for matched commands.

The hook fires for BOTH mainthread and subagent-dispatched tool calls (preflight #3 confirmed). Subagent invocations are distinguishable by the `agent_id`/`agent_type` envelope fields — used for attribution in trip-wire reports.

Env-gated: the hook body checks `SPINE_QUEUE=1` and exits 0 (no-op) otherwise. This keeps interactive sessions unaffected.

**Fail-secure**: if `profile.json` is missing OR the hook binary is missing OR not executable, the supervisor refuses to start.

## `profile.json` Shape

Generated by the run-queue skill from `queue.yaml` (Slice D) or hand-written. Consumed by (a) `claude -p --settings profile.json` (registers hooks) and (b) the supervisor (extracts allow/deny lists for flag construction).

```json
{
  "schema_version": "1",
  "allowed_tools": ["Bash", "Edit", "Write", "Read", "Glob", "Grep", "Task"],
  "disallowed_tools": ["WebFetch", "WebSearch"],
  "patterns": {
    "deny": [
      { "match": "Bash", "command_prefix": "git push",  "reason": "remote write outside queue scope" },
      { "match": "Bash", "command_prefix": "git -C ",   "reason": "out-of-project git operation" },
      { "match": "Bash", "command_regex":  "^rm\\s+-[a-zA-Z]*[rR]", "reason": "use trash(1) — recursive rm blocked" },
      { "match": "Bash", "command_regex":  "\\bcurl\\b.*(-T|-F|-d @|--upload-file)", "reason": "curl file upload" },
      { "match": "Bash", "command_prefix": "npm publish", "reason": "package publish" }
    ]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [ { "type": "command", "command": "hooks/guard-queue-shell.sh" } ]
      }
    ]
  }
}
```

## Pattern matching semantics

Each `patterns.deny[]` entry matches a tool invocation when:

1. `match` equals the tool name (`Bash`, `Edit`, `Write`, ...), AND
2. Exactly one of `command_prefix`, `command_regex`, `path_prefix`, `path_regex` matches the relevant field of `tool_input`.

Field routing:

| `match` | Field inspected |
|---------|-----------------|
| `Bash`  | `tool_input.command` (after normalization: collapse whitespace, strip leading `rtk ` proxy) |
| `Edit`, `Write` | `tool_input.file_path` |
| `Read`, `Glob`, `Grep` | not deny-listed in v1 (read-only) |

Normalization is fixed — patterns work against the normalized form.

## Out-of-project writes

The supervisor additionally constrains Edit/Write to the project tree: any `tool_input.file_path` not resolving under the queue's repo root triggers an out-of-project deny. Implementation sits in `guard-queue-shell.sh`; configured via implicit rule, not via `patterns.deny`.

## Trip-wire behavior

On ANY deny:

1. Hook writes a structured line to `<queue-dir>/queue-log.md` (tool, command, reason, `agent_id` if subagent).
2. Hook writes `<queue-dir>/WOKE-ME-UP.md` with task id, timestamp, deny reason. First deny wins — subsequent denies append.
3. Hook emits `permissionDecision: deny` JSON + exits 2 — the tool call is blocked.
4. The child `claude -p` reports the block to its session; supervisor sees non-`complete` status at terminal check.
5. Supervisor halts the queue (current task and all pending tasks), preserves the task's local branch for inspection.

The design deliberately halts on ANY trip-wire, not just git-push — the cost of a false negative overnight is higher than a false positive (user decides in the morning whether the block was correct).

## Default Profile (Slice A)

For v1 Slice A, the default profile emitted when `queue.yaml` does not reference a `profile.json` is conservative:

- `allowed_tools`: `Bash`, `Edit`, `Write`, `Read`, `Glob`, `Grep`, `Task`
- `disallowed_tools`: `WebFetch`, `WebSearch`
- `patterns.deny`: git push, `git -C`, recursive rm (falls back to `trash`), curl uploads, `npm publish`, `pnpm publish`, `yarn publish`, `cargo publish`

Users override by writing their own `profile.json` and pointing `queue.yaml.profile` at it.
